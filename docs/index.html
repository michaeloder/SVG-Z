Chapter 01  ·  svgz.js v1.0

What is SVG-Z?

SVG changed the web by making graphics declarative. You describe what you want — a circle here, a path there — and the browser draws it. No pixel-pushing, no render loops, no matrix math leaking into your markup.

SVG-Z asks: what if you could do the same thing in three dimensions?

Not with a full 3D engine. Not with meshes, shaders, or normal maps. Just the same clean declarative approach you already know from SVG — extended with a Z coordinate.

Every path command gains a third number. Every shape can exist at any depth. A z-buffer resolves which geometry is in front. Transparent surfaces blend correctly. The camera is a single XML element. And the result renders to a plain <canvas> — no WebGL required.

The core idea

In regular SVG you write:

SVG — flat, 2D
<svg width="200" height="200">
  <circle cx="100" cy="100" r="60" fill="#7c6aff"/>
</svg>

In SVG-Z you write the same thing — but each shape has a position in 3D space. Orbit the camera and watch them correctly occlude each other from every angle:

SVG-Z — pyramid, sphere and cylinder in 3D

Drag to orbit the camera. Scroll to zoom. The depth buffer resolves every occlusion automatically — no painter's algorithm, no draw-order tricks.

What SVG-Z is not

SVG-Z is deliberately not a full 3D engine. There is no lighting model. No materials. No textures. No shadows. The rendering is flat — colour comes from the fill and stroke attributes, just like SVG. What you gain is correct 3D positioning, surfaces of revolution, Bézier patches, and per-pixel depth — without any of the complexity of a scene graph engine.

Think of it as the gap between SVG and Three.js. It lives happily in that space.

Chapter 02

Paths & Polygons

Before SVG-Z had surface primitives, it had path3d — the 3D analogue of SVG's <path>. The same command letters, the same structure, but every coordinate is an x,y,z triple instead of an x,y pair.

The d3 attribute

Where SVG uses d="M 10,10 L 100,50", SVG-Z uses d3="M 10,10,0 L 100,50,20". The third number is the Z coordinate — depth in world space.

path3d — a 3D spiral

Orbit the view to see the curves in 3D. Each path uses a different stroke width and colour — these are independent primitives, each with its own depth relative to the z-buffer.

Closed paths and fills

End a path with Z to close it back to the M point. Closed paths can be filled — the interior depth is interpolated from the boundary using centroid fan triangulation.

Chapter 03

The Camera

The <camera> element defines where you're standing, what you're looking at, and how wide your lens is. SVG-Z uses a standard perspective projection — the same model used in OpenGL, WebGL, and virtually every 3D engine.

eye, center, up

These three vectors fully specify the camera's position and orientation in world space:

  • eye — where the camera is positioned
  • center — the point it looks toward
  • up — which direction is "up" (usually 0,1,0)
camera — eye and center

Field of view

The fov attribute controls the vertical field of view in degrees. A low value (15–30°) creates a telephoto look — objects appear compressed and the perspective is flat. A high value (70–100°) creates a wide-angle look with strong perspective distortion.

camera — fov comparison

Try fov="20" — the spheres appear similar in size regardless of depth. Now try fov="80" — the near spheres loom much larger. This is perspective foreshortening in action.

The orbit camera

SVG-Z layers an interactive orbit system on top of the scene camera. The scene camera sets the starting position; the orbit controls modify it at runtime. The orbit is always centred on the center point.

Chapter 04

Depth & the Z-Buffer

Most 2D graphics systems resolve what's in front using painter's algorithm — draw everything back to front, later draws overwrite earlier ones. It works for non-intersecting shapes, but breaks the moment two surfaces cross.

SVG-Z uses a z-buffer: a per-pixel depth value stored alongside the colour. Every time a pixel is about to be drawn, its depth is compared to whatever is already there. If it's closer, it wins. If not, it's discarded. The result is that intersection is free — the depth buffer resolves it automatically.

depth-bias

Coplanar surfaces — two polygons at exactly the same Z — cause z-fighting: the depth values are so close that rounding errors cause noisy flickering. The depth-bias attribute nudges a surface's depth by a small NDC offset to resolve this:

depth-bias — resolving coplanar surfaces

render-order

For cases where depth alone isn't enough — typically when you want one element to always render on top of another regardless of depth — use render-order. Elements are sorted by this integer value first, then by depth. Higher numbers render later (on top).

Lathe

A lathe in woodworking spins a piece of wood against a cutting tool, producing a surface of revolution. <lathe3d> does the same digitally — you draw a 2D profile curve in the d2 attribute and the renderer revolves it around an axis.

Chapter 09

Bézier Patches

All the surface primitives so far — spheres, cylinders, lathe, extrude — are procedural. You set parameters and the geometry is generated. But sometimes you need a surface that doesn't fit any procedure. A saddle shape. A twist. A wave.

For these, SVG-Z has bicubic Bézier patches.

The B command

The <surface3d> element takes a d3 attribute containing one or more B commands. Each B is followed by 16 control points in a 4×4 grid. The four corner points lie exactly on the surface; the interior points act as tangent handles.

surface3d — saddle surface

The saddle dips down at the front and back edges and rises at the left and right — the classic hyperbolic paraboloid shape. Orbit to see it from different angles.

Multiple patches — twisted ribbon

A single d3 attribute can contain multiple B commands. Here a single patch twists from a vertical cross-section at the left end to a horizontal one at the right:

Chapter 10

Transparency

Transparency in 3D is a hard problem. The naive approach — draw everything back-to-front — fails the moment two transparent surfaces intersect. There is no single correct sort order when surfaces cross.

SVG-Z uses Weighted Blended Order-Independent Transparency (WBOIT), a technique that accumulates transparent fragments without sorting and resolves them in a single compositing pass.

How it works

Every frame runs three passes. First, all opaque geometry is rasterised normally into the z-buffer. Then, transparent geometry is accumulated into two per-pixel buffers — a weighted colour sum and a running product of (1 − alpha) values called the revealage. Finally, a resolve pass composites the weighted result over the opaque framebuffer:

weight(z, α) = α · clamp(0.03 / (ε + z⁴), 0.01, 3000)
output = weightedColour × (1 − revealage) + opaqueColour × revealage
transparency — WBOIT intersection test

Orbit the camera to any angle. The three plane intersections always blend correctly — the central white sphere always appears through all three, and the blend colours at each intersection are stable regardless of view direction.

// note
Any element with opacity < 1, or a fill/stroke colour with alpha < 255, is automatically routed through the WBOIT pass. Fully opaque elements always use the standard z-buffer path.
// exercises

Chapter 11

Anti-Aliasing

Rasterisation produces aliased edges — the staircase effect where diagonal lines appear jagged. SVG-Z offers two anti-aliasing approaches with very different cost/quality profiles.

TSAA — Temporal Super-Sampling

Each frame, the projection matrix is offset by a tiny sub-pixel amount using a Halton sequence — a low-discrepancy pattern that distributes 8 sample positions very evenly across the pixel footprint. The result is blended into a history buffer:

history = current × blendAlpha + history × (1 − blendAlpha)

When the camera is still, blendAlpha = 0.2 — the history accumulates slowly over 8 frames. When the camera moves, blendAlpha = 0.6 — the history turns over quickly to prevent ghosting.

MSAA — Multisample Anti-Aliasing

MSAA renders into a 2× or 4× supersampled internal buffer and box-filters down to the display resolution. The result is consistent in motion (no history to become stale), but costs 4× or 16× fill rate. The demos in this book use MSAA×4 as the default.

See the difference between modes:

Anti-aliasing OFF — jagged edges visible
MSAA×4 — smooth edges
TSAA — temporal smoothing (wait a moment when still)

TSAA and MSAA×4 produce similar quality at rest. The key difference: TSAA costs nothing extra per frame but lags slightly after motion; MSAA×4 is sharp immediately but costs 16× more rasterisation.

Setting AA in the scene

You can set the anti-aliasing mode directly inside the <camera> element rather than relying on the embedding page:

aa — declared in camera element
<camera fov="50" eye="3,3,18" center="0,0,0" up="0,1,0"
       aa="msaa4"/>

This is the recommended approach for standalone <svgz> embeds — the scene carries its own quality setting rather than depending on how it's hosted.

Chapter 01

Controls & Interaction

Every SVG-Z canvas is interactive. You can orbit the camera, zoom in and out, and pan the view — all without writing any JavaScript. The controls are built in and work on any rendered canvas.

Orbit (rotate)

Left-drag on the canvas to orbit the camera around the scene's center point. The horizontal drag changes the azimuth angle; vertical drag changes the elevation. The camera always keeps the up vector as its reference.

Zoom

Scroll wheel to zoom in and out. This adjusts the orbit radius — the camera moves closer to or further from the center point along the current view direction. The minimum zoom keeps the camera at half the original distance.

Pan

Shift + left-drag, or right-drag, to pan the view. Panning moves both the camera eye and the center together, shifting the entire scene without changing the viewing direction.

The controls button

Each canvas has a small ? button in the bottom-right corner. Click it to show a quick-reference overlay. Try it now:

controls — try the ? button on the canvas

Keyboard shortcut

Press Ctrl+Enter (or Cmd+Enter on Mac) while the code editor is focused to re-run the scene without clicking the RUN button.

Auto-rotation

Add orbit="1" to a <camera> to make a scene auto-rotate. Positive values spin forward; negative values reverse. The speed is a multiplier — orbit="2" is twice as fast. Dragging to orbit manually always takes over immediately.

orbit — built into the camera element
